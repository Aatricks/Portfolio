`llmedge` is more interesting as an Android runtime layer than as a feature list. The codebase is organized around a thin `LLMEdge` facade, a separate model-resolution path, and pooled native runtimes that stay explicit about backends, memory, and ownership.

- `LLMEdge` exposes text, speech, image, vision, and RAG clients behind one lifecycle-aware scope instead of forcing each feature to own native state independently
- `ModelRepository` and the Hugging Face download pipeline keep artifact selection, validation, and caching separate from inference
- the runtime stack uses `RuntimePool` and `RuntimeCoordinator` to acquire engines and retry across OpenCL, Vulkan, and CPU when a backend fails
- the library is not just JNI glue: the repo also carries JVM tests, Android instrumentation, and opt-in native E2E coverage for text, speech, image, and video flows

That mix of Kotlin API design, Android runtime policy, and native-engine orchestration is what makes the repo representative of the work I want to keep doing.
